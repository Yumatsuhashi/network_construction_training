# Steering

`.steering/` を作業記録の正本として扱い、実際の進捗と記録を一致させる。

## 作業記録を選択する

1. `.steering/*/tasklist.md` を調べ、今回の依頼と同じ作業の未完了記録があるか確認する。
2. 同じ作業があれば新規作成せず、その `requirements.md`、`design.md`、`tasklist.md` を読んで再開する。
3. 無関係な未完了記録は変更しない。
4. 新しいまとまった作業なら、現在日付と短い作業名から `.steering/[YYYYMMDD]-[作業名]/` を作成する。
5. 説明だけの質問、読み取りだけの確認、単一ファイルの軽微な修正では、新しい記録を強制しない。

Plan Mode などファイル変更が禁止されている状態では記録を作成せず、実装が許可されてから作成する。

## 新しい記録を作成する

1. `要件/`、`設計書_md/`、`設問/` と今回の入力を読み、関係する正本を特定する。
2. `.steering/templates/requirements.md`、`design.md`、`tasklist.md` を新しい作業ディレクトリへ展開する。
3. すべてのプレースホルダーを今回の具体的な内容へ置き換える。
4. requirements に目的、受け入れ条件、スコープ外、参照資料を記録する。
5. design にアプローチ、変更対象、制約、検証方法、実施順序を記録する。
6. tasklist には今回実行する検証可能なタスクだけを依存順に記録する。将来候補や未確定の案は含めない。

## tasklist に従って実施する

1. 作業開始前に tasklist を読み、次の未完了タスクを1つ特定する。
2. 必要に応じて Codex のセッション内計画も更新する。ただし tasklist を正式な進捗源とする。
3. `要件/`、`設計書_md/`、`設問/` の根拠と整合させて作業する。
4. タスク完了直後に `apply_patch` で該当する `[ ]` を `[x]` にする。サブタスクを個別に更新し、親タスクは全サブタスク完了後に更新する。
5. 作業中に方針や対象が変わったら、先に requirements、design、tasklist を実態へ合わせる。
6. タスクが大きすぎる場合は実行可能なサブタスクへ分割し、1つずつ完了させる。
7. 不要になったタスクは削除せず、取り消し線と具体的な理由を付けて完了扱いにする。

複数タスクを終えてからまとめて進捗を更新しない。

## 中断と再開を記録する

次の場合は未完了タスクを残して中断できる。

- ユーザーの設計判断や追加情報を待つ場合
- 実機、シミュレータ、外部サービスなど必要な環境が利用できない場合
- 権限や技術的制約により実行できない場合
- ユーザーが中断を明示した場合

中断時は tasklist の「中断・再開情報」に次を記録する。

- 状態
- 最後に完了したタスクと次の未完了タスク
- 中断した具体的な理由
- 再開時に最初に確認・実行する内容

完了していないタスクを `[x]` にせず、振り返りを完了扱いで書かない。再開時は3文書と中断情報を読み、実際の成果物との差分を確認してから続行する。

## 完了と振り返りを記録する

1. tasklist に未完了タスクがないことを確認する。
2. 受け入れ条件と設計上の検証方法を実行する。
3. 永続ドキュメントを訂正した場合は、変更理由を tasklist に記録する。
4. 実施完了日、計画との差分、技術的な知見、プロセス改善、次回への提案を振り返りへ記録する。
5. 「中断・再開情報」の状態を `完了` にし、再開手順を `なし` にする。
6. 最終回答で成果物、検証結果、残存する制約を簡潔に報告する。
