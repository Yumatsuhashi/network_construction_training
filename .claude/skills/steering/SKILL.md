# Steering スキル

`.steering/` を作業記録の正本として扱い、実際の作業と tasklist の状態を一致させる。

## 作業記録の選択

1. `.steering/*/tasklist.md` を確認する。
2. 今回と同じ作業の未完了記録があれば、その `requirements.md`、`design.md`、`tasklist.md` を読んで再開する。
3. 無関係な未完了記録は変更しない。
4. 新しいまとまった作業なら `.steering/[YYYYMMDD]-[作業名]/` を作成する。
5. 説明だけの質問、読み取り確認、単一ファイルの軽微な修正では記録作成を強制しない。

## 新しい記録の作成

`要件/`、`設計書_md/`、`設問/` の該当箇所を読んだうえで、次の共通テンプレートを展開する。

- `.steering/templates/requirements.md` → 作業ディレクトリの `requirements.md`
- `.steering/templates/design.md` → 作業ディレクトリの `design.md`
- `.steering/templates/tasklist.md` → 作業ディレクトリの `tasklist.md`

プレースホルダーを具体化し、requirements に目的・受け入れ条件・スコープ外、design にアプローチ・変更対象・検証方法、tasklist に今回実行するタスクだけを記録する。

## 作業実施と進捗更新

1. 作業前に tasklist を読み、次の未完了タスクを1つ特定する。
2. TaskCreate/TaskUpdate は補助的に利用し、tasklist を正式な進捗源とする。
3. 要件、設計書、設問と整合させて作業する。
4. 完了直後に Edit で該当する `[ ]` を `[x]` にする。サブタスクを個別に更新し、親は全サブタスク完了後に更新する。
5. 方針や対象が変わったら、requirements、design、tasklist を先に実態へ合わせる。
6. 大きすぎるタスクは実行可能なサブタスクへ分割する。
7. 不要になったタスクは削除せず、取り消し線と具体的な理由を付けて完了扱いにする。

複数タスクを終えてからまとめて進捗を更新しない。

## 中断と再開

次の場合は未完了タスクを残して中断できる。

- ユーザーの判断や追加情報を待つ場合
- 実機、シミュレータ、外部サービスなど必要な環境が利用できない場合
- 権限や技術的制約により実行できない場合
- ユーザーが中断を明示した場合

中断時は tasklist の「中断・再開情報」に状態、現在地、具体的な理由、再開手順を記録する。未完了タスクを `[x]` にせず、完了扱いの振り返りを書かない。再開時は3文書と実際の成果物との差分を確認してから続行する。

## 完了と振り返り

1. tasklist に未完了がないことを確認する。
2. requirements の受け入れ条件と design の検証方法を確認する。
3. 実施完了日、計画との差分、学んだこと、次回への改善提案を tasklist に記録する。
4. 中断・再開情報を完了状態にする。
5. 永続ドキュメントを訂正した場合は、その変更理由も記録する。
6. ユーザーへ成果物、検証結果、残存する制約を報告する。
