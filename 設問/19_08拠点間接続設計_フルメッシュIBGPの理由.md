# 設問：拠点間接続設計③ ①（出典：設計書_md/08_拠点間接続設計.md）

> 出典：①ネットワーク要件・詳細設計書(TC).pptx スライド53

## 該当する要件
- 拠点内はループバックインターフェースを使用し、**フルメッシュでIBGPネイバーを接続する**（拠点内は認証不要）
- ただし、DSWやCSWのBGPテーブルに192.168のルートが存在しないこと

## 【設問】
> 拠点内はフルメッシュでIBGPネイバーを接続する理由は何ですか？

## 回答

### 理由：IBGPの「スプリットホライズン的ルール」により、フルメッシュ以外では経路が全ノードに伝わらないため

BGPには「IBGP経由で学習した経路は、別のIBGPピアへは再広告しない」というループ防止ルールがある（eBGPで学習した経路はIBGPピアへ広告されるが、IBGPで学習した経路をさらに他のIBGPピアへ中継することはデフォルトでは行わない）。

本設計ではR1・R2は同一AS（65001）に属しており、拠点内のR1, R2, CSW1, CSW2, DSW1, DSW2はすべて **IBGP** 関係になる。もしフルメッシュでなく、例えば「R1-CSW1-DSW1」のような直列（部分的）なIBGP接続にしてしまうと、

- R1がIP-VPN網（eBGP）から学習した拠点2・拠点3宛の経路をCSW1へは広告できても
- CSW1はそれをさらにDSW1やCSW2へIBGP経由で再広告できない

という「経路の伝播漏れ」が発生し、DSW1やCSW2は拠点間通信の経路を一切知ることができなくなってしまう。

これを回避する方法は主に3つある。
1. **フルメッシュIBGP**（全ノードが直接ピアを張る）
2. ルートリフレクタ（Route Reflector）
3. コンフェデレーション（BGP Confederation）

本設計は6台（R1, R2, CSW1, CSW2, DSW1, DSW2）という比較的小規模な構成であり、ルートリフレクタ等の追加設計を導入せずシンプルに実現できるため、**フルメッシュIBGP** が採用されている。

### 設定イメージ（ループバックアドレスを用いたIBGPフルメッシュ、R1の例）

```
R1(config)# router bgp 65001
R1(config-router)# neighbor 10.1.254.6 remote-as 65001
R1(config-router)# neighbor 10.1.254.6 update-source Loopback0
R1(config-router)# neighbor 10.1.254.3 remote-as 65001
R1(config-router)# neighbor 10.1.254.3 update-source Loopback0
R1(config-router)# neighbor 10.1.254.4 remote-as 65001
R1(config-router)# neighbor 10.1.254.4 update-source Loopback0
R1(config-router)# neighbor 10.1.254.1 remote-as 65001
R1(config-router)# neighbor 10.1.254.1 update-source Loopback0
R1(config-router)# neighbor 10.1.254.2 remote-as 65001
R1(config-router)# neighbor 10.1.254.2 update-source Loopback0
```
（R2, CSW1, CSW2, DSW1, DSW2についても、自分以外の全ノードへ同様にneighbor文を設定する＝合計6台の完全グラフ＝15本のIBGPセッション）

要件通り「拠点内は認証不要」のため `password` は設定しない（IP-VPN側のeBGPネイバーのみ認証パスワードcisco123を使用）。

### なぜループバックを使うのか
物理インターフェースのIPをBGPピアアドレスにすると、リンク自体が生きていてもOSPFの迂回経路が使える場面でBGPセッションだけが切れてしまうことがある。ループバックは特定の物理リンクに依存しない「常に生きている」アドレスであるため、複数経路（OSPFでマルチパス）がある構成ではピアリングにループバックを使うのが定石。

### 補足
- 「DSWやCSWのBGPテーブルに192.168のルートが存在しないこと」という条件は、R1/R2とIP-VPN網の間の接続用セグメント（192.168.11.0/30, 192.168.12.0/30）をBGPの `network` 文などでわざわざ内部へ広告しない（redistributeしない）ことを意味する。これらは単なる物理リンクのアドレッシングであり、拠点内の経路制御には不要なため、意図的にBGPテーブルへ含めない設計になっている。
