# AGENTS.md

## プロジェクト概要

このリポジトリは Cisco ネットワーク構築実習の設計検討、コンフィグ作成、トラブルシュート、設問回答を管理する。ソフトウェア開発プロジェクトではないため、一般的な lint、test、build は存在しない。検証では要件、設計、設問、Cisco config の整合性を重視する。

## ドキュメントの役割

- `要件/`: 元となる会社概要、ネットワーク要件、物理設計の要求メモ
- `設計書_md/`: 01〜09の設計書と物理・論理構成図。設計パラメータの正本
- `設問/`: 演習の設問と回答。採用するコマンドや理由の根拠
- `config/`: カテゴリ別・機器別 Cisco config、確認コマンド、統合 config の成果物
- `.steering/`: 作業単位の要求、設計、タスクリスト、判断履歴、振り返り

永続的な参照ドキュメントと作業記録を混同しない。要件や設計そのものを訂正する必要がある場合だけ正本を更新し、その理由を作業中の tasklist に記録する。

## Steering の運用

複数工程の config 作成、設計検討、トラブルシュート、設問回答、既存作業の再開では、リポジトリ skill `steering` を使用する。明示的に呼ぶ場合は `$steering` を指定する。

- 今回の依頼と同じ未完了記録が `.steering/` にあれば、その3文書を読んで再開する。
- 無関係な未完了記録や成果物を変更しない。
- 新規のまとまった作業では `.steering/[YYYYMMDD]-[作業名]/` を作成する。
- `.steering/templates/` の requirements、design、tasklist を使用する。
- tasklist を正式な進捗源とし、完了したタスクをその都度更新する。
- 正当な中断時は未完了を完了扱いにせず、理由、現在地、再開手順を記録する。
- 説明だけの質問、読み取り確認、単一ファイルの軽微な修正には新しい記録を強制しない。
- Plan Mode など変更禁止の状態では記録を作らず、実装開始後に作成する。

## 作業原則

- 変更前に関係する要件、設計書、設問、既存 config を読む。
- IPアドレス、サブネット、VLAN、インターフェイス、AS番号、ルーティング、冗長化、認証、ACL の値を推測しない。正本間で矛盾した場合は明示する。
- config は既存のカテゴリ構造と機器別ファイル構成に合わせる。
- 設問の例を実トポロジへ読み替えた場合は、読み替えた対象と理由を記録する。
- Cisco IOS のバージョン差が動作に影響する場合は、前提と互換性を明記する。
- 確認コマンドには、実行する show コマンドだけでなく確認すべき状態も記載する。
- ユーザーの既存変更を保持し、依頼と無関係な差分を戻さない。

## 検証

- IPアドレスとサブネットを `設計書_md/03_論理設計.md` の表と突き合わせる。
- HSRP、STP、EtherChannel、OSPF、BGP、ACL、管理設定を該当設計書と設問に突き合わせる。
- カテゴリ別 config と統合 config の重複、欠落、矛盾を確認する。
- 実機やシミュレータを利用できない場合は静的検証と動作未確認の範囲を区別して報告する。
- 作業完了時に `git diff` と `git status` を確認し、変更範囲を明示する。
