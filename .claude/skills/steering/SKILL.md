# Steering スキル

ステアリングファイル（`.steering/`）に基づいて演習の作業を進め、tasklist.mdの進捗管理を確実に行うスキルです。ネットワーク構築実習（設計検討・コンフィグ作成・トラブルシュート・設問回答など、コードを書かない作業全般）向けに調整されています。

## スキルの目的

- ステアリングファイル（requirements.md, design.md, tasklist.md）の作成支援
- tasklist.mdに基づいた段階的な作業管理
- **進捗の自動追跡とtasklist.md更新の強制**
- 作業完了後の振り返り記録

## 使用タイミング

このスキルは以下のタイミングで使用してください：

1. **作業計画時**: ステアリングファイルを作成する時（例：新しい設問群への回答、機器のコンフィグ作成、トラブルシュートの実施など、まとまった作業を始める時）
2. **作業実施時**: tasklist.mdに従って作業する時
3. **振り返り時**: 作業完了後の振り返りを記録する時

## モード1: ステアリングファイル作成

### 目的
新しい作業のためのステアリングファイルを作成します。

### 手順

1. **ステアリングディレクトリの確認**
   ```
   現在の日付を取得し、`.steering/[YYYYMMDD]-[作業名]/` の形式でディレクトリを作成
   ```

2. **永続的な参照ドキュメントの確認**
   - `要件/` … 元となる要求メモ（会社概要・ネットワーク要件・物理設計等）
   - `設計書_md/` … pptxから起こした設計書（物理設計・論理設計・耐障害性・可用性・セキュリティ・管理・拠点間接続設計・オプション）と構成図（物理構成図.png / 論理構成図.png）
   - `設問/` … 演習の設問と回答一覧

   これらを読んで、今回の作業が設計書・要件のどの部分に関わるかを理解する

3. **テンプレートからファイル作成**

   以下のテンプレートを読み込み、プレースホルダーを具体的な内容に置き換えてファイルを作成:

   - `.claude/skills/steering/templates/requirements.md` → `.steering/[日付]-[作業名]/requirements.md`
   - `.claude/skills/steering/templates/design.md` → `.steering/[日付]-[作業名]/design.md`
   - `.claude/skills/steering/templates/tasklist.md` → `.steering/[日付]-[作業名]/tasklist.md`

4. **tasklist.mdの詳細化**

   requirements.mdとdesign.mdに基づいて、tasklist.mdを詳細化:
   - 各フェーズのタスクを具体的に記述
   - サブタスクも明確に
   - 作業の順序を明記

## モード2: 作業実施（最重要）

### 目的
tasklist.mdに従って作業を進め、**進捗を確実にドキュメントに記録**します。

### 🚨 重要な原則

**MUST（必須）**:
- tasklist.mdを常に開いた状態で作業
- タスク完了時に必ずEditツールで`[ ]`→`[x]`に更新
- **tasklist.mdの全タスクが完了するまで作業を継続する**
- NEVER: tasklist.mdを更新せずに次のタスクに進まない

**NEVER（禁止）**:
- tasklist.mdを見ずに作業を進める
- TaskCreate/TaskUpdateツールだけで進捗管理する（これらは本環境のタスク管理機能で補助的な位置づけ、tasklist.mdが正式）
- 複数タスクをまとめて更新する（リアルタイムに更新する）
- **「時間の都合により」「別作業として実施予定」などの理由でタスクをスキップする**
- **未完了タスク（`[ ]`）を残したまま作業を終了する**

### 🚨 タスク完全完了の原則

**絶対に守るべきルール**:

1. **tasklist.mdの全タスクが完了するまで作業を継続すること**
   - 全てのタスクが`[x]`になるまで作業を継続
   - 「時間がかかりすぎる」「難しい」などの理由でスキップしない
   - 未完了タスクがある状態で振り返りを書かない

2. **タスクスキップは原則禁止**
   - 「時間の都合により別タスクとして実施予定」は禁止
   - 「複雑すぎるため後回し」は禁止
   - スキップが許可されるのは技術的な理由のみ（下記参照）

3. **タスクが大きすぎる場合の対処法**
   - タスクを小さなサブタスクに分割する
   - 分割したサブタスクをtasklist.mdに追加
   - サブタスクを1つずつ完了させる

4. **技術的な理由でタスクが不要になった場合のみスキップ許可**

   以下の理由に該当する場合のみスキップ可能:
   - 要件・設計の変更により、そのタスク自体が不要になった
   - 機材・環境上の制約により、タスクが実行不可能になった
   - 上位の設計変更により、このタスクが無意味になった

   スキップ手順:
   - tasklist.mdに理由を明記してスキップマークを付ける
   - 例: `- [x] ~~タスク名~~（要件変更により不要: 具体的な理由）`
   - 振り返りセクションに変更理由を詳細に記録

### 作業フロー

#### ステップ1: tasklist.mdを読み込む

```
Read('.steering/[日付]-[作業名]/tasklist.md')
```

全体のタスク構造を把握し、次に着手すべきタスクを特定する。

#### ステップ2: TaskCreateでタスク管理開始

tasklist.mdの内容に基づいてTaskCreateツールでタスクを登録する:
- これは本環境の補助的なタスク管理（進捗の可視化用）
- **tasklist.mdこそが正式なドキュメント**

#### ステップ3: タスクループ（各タスクで繰り返す）

**3-1. 次のタスクを確認**
```
tasklist.mdを読み、次の未完了タスク（`[ ]`）を特定
```

**3-2. TaskUpdateでステータス更新**
```
TaskUpdateツールで該当タスクを"in_progress"に変更
```

**3-3. 作業を実行**
```
要件/・設計書_md/・設問/ の内容と整合させながら作業を実施
（コンフィグ作成、設問への回答、構成の検討、トラブルシュート等）
```

**3-4. タスク完了をtasklist.mdに記録（必須）**
```
作業完了後、必ずEditツールでtasklist.mdを更新して完了を記録

サブタスクがある場合はサブタスクも個別に更新する
```

**3-5. TaskUpdateでもステータス更新**
```
TaskUpdateツールで該当タスクを"completed"に変更
```

**3-6. 次のタスクへ**
```
ステップ3-1に戻る
```

#### ステップ4: フェーズ完了時の確認

各フェーズが完了したら:

1. **tasklist.mdを読み込んで進捗確認**
2. **完了したタスクを確認**（すべてのタスクが`[x]`になっているか、見落としがないか）
3. **ユーザーに報告**（「フェーズ◯◯が完了しました。tasklist.mdの進捗を確認してください。」）

#### ステップ4.5: 全タスク完了チェック（必須）

**全フェーズの作業完了後、振り返りを書く前に必ず実行**:

1. tasklist.mdを読み込む
2. 未完了タスク（`[ ]`）がないか確認
3. 未完了タスクが見つかった場合
   - **❌ やってはいけないこと**: 「時間の都合により」と振り返りに書いて無視する
   - **✅ 正しい対処法**:
     - パターンA: タスクループに戻り実装する
     - パターンB: タスクが大きすぎる場合はサブタスクに分割して1つずつ完了させる
     - パターンC: 技術的な理由でタスクが不要になった場合のみ、理由を明記してスキップする
4. 全タスク完了を確認できた場合のみ次へ

#### ステップ5: 全タスク完了後

1. **最終確認**: 全タスクが`[x]`になっていることを確認
2. **振り返りセクションに記録**（作業完了日／計画と実績の差分／学んだこと／次回への改善提案）

## モード3: 振り返り

### 目的
作業完了後、tasklist.mdに振り返りを記録します。

### 手順

1. tasklist.mdを読み込む
2. 振り返り内容を作成（作業完了日、計画と実績の差分、学んだこと、次回への改善提案）
3. Editツールでtasklist.mdの「実装後の振り返り」セクションを更新
4. ユーザーに報告（「振り返りをtasklist.mdに記録しました。内容を確認してください。」）
5. 今回の作業が `要件/` や `設計書_md/`（永続的な参照ドキュメント）の内容に影響・訂正を与える場合は、該当ファイルもEditツールで更新する

## トラブルシューティング

### tasklist.mdの更新を忘れた場合

1. 即座に更新を実行（Read → 完了タスクを特定 → Editツールで`[x]`に更新）
2. ユーザーに報告（「tasklist.mdの更新が遅れていたため、現在の進捗を反映しました。」）
3. 再発防止：次のタスクから確実に更新する

### tasklist.mdと実際の作業内容の乖離

計画と実際が大きく異なる場合:

1. tasklist.mdに注釈を追加（`- [x] タスク名（方針を変更: 理由）`）
2. 必要に応じて新しいタスクを追加
3. 変更が大きい場合はdesign.mdも更新

## チェックリスト（最重要）

作業前に必ず確認:
- [ ] tasklist.mdを読み込んだか？
- [ ] 次のタスクを特定したか？

作業後に必ず確認:
- [ ] タスク完了時にEditツールで更新したか？
- [ ] tasklist.mdの進捗を確認したか？
- [ ] ユーザーが見て進捗が分かる状態か？

## スキルの効果

このスキルを正しく使用すると:

- ✅ tasklist.mdが常に最新の進捗を反映
- ✅ ユーザーが進捗を一目で把握できる
- ✅ 演習の記録と実際にやったことの乖離がなくなる
- ✅ 振り返りが容易になり、次回の改善につながる
- ✅ 演習の履歴として価値がある記録が残る

## 重要なリマインダー

🚨 **このスキルの最も重要な役割は、tasklist.mdの進捗管理を確実に行うことです。**

- TaskCreate/TaskUpdateは補助的な進捗表示（セッション内限定）
- **tasklist.mdこそが永続的なドキュメント（ユーザーが見る記録）**

作業中は常に「ユーザーがtasklist.mdを見たときに進捗が分かるか？」を自問してください。
